Author: Gazettm Date: 十二月 20, 2025  0:33:04 Category: CSAPP

CSAPP_ECF

ECF，即为Exceptional Control Flow，CSAPP中文书中将其翻译为异常控制流。

指令在内存中排序，PC寄存器依次执行指令，这就是最基础的控制流。

异常

系统需要能够对系统状态的变化做出反应。现代系统通过使控制流发生突变来对这些情况做出反应。

当应用程序遇到一个异常时，会跳转到异常处理程序，若能够处理，则返回，若不能处理，则终止该程序。

异常处理

系统中可能的每种类型的异常都分配了一个唯一的非负整数的异常号。其中一些号码是由处理器的设计者分配，其他号码是由操作系统内核的设计者分配。前者包含被零除，缺页，内存访问异常，断点以及算术运算溢出，后者包括系统调用和来自外部I/O设备的信号。在系统启动时（计算机重启或者加电），操作系统分配和初始化一张称为异常表的跳转表，该跳转表包含某个异常的异常处理程序的地址。异常表的起始地址放在一个叫做异常表基址寄存器的特殊CPU寄存器里。

异常类似于过程调用，但也有一些重要的不同之处

  • 过程调用的时候，在跳转到处理程序之前，处理器将返回地址压入栈中。然而根据异常类型，返回地址可能是当前地址或者当前直流的下一条指令
  • 处理器也会把一些额外的处理器状态压入栈中，在处理程序返回的时候，重新开始执行被中断的程序会需要这些状态。（比如x86-64的系统会将包含当前条件码的EFLAGS寄存器和其他内容压入栈中
  • 如果控制从用户程序转移到内核中，所有这些项目都被压入到内核栈，而不是用户栈（关于内核与用户，稍后会讲
  • 异常处理程序运行在内核模式下，意味着他们对所有的系统资源都有完全的访问权限。

异常的类型

类型 原因 异步/同步 返回行为
中断 来自I/O设备的信号 异步 总是返回到下一条指令
陷阱 有意的异常 同步 总是返回到下一条指令
故障 潜在可恢复的错误 同步 可能返回到当天指令
终止 不可恢复的错误 同步 不会返回

中断

中断时异步发生的，是来自外部的I/O设备的信号的结果。中断并非由一条专门的指令造成，从这个意义上来说是异步的。

以下是中断异常处理程序，从当前指令返回到下一条指令。

  1. 当前指令执行过程中，中断引脚电压升高
  2. 当前指令完成后，控制传递给处理程序
  3. 中断处理程序运行
  4. 处理程序返回下一条指令

陷阱和系统调用

陷阱是有意的异常。最重要的用途是在用户程序与内核之间提供一个像过程一样的接口，叫做系统调用

用户经常需要向内核请求服务，比如读取一个文件，创建一个新的进程，加载一个新的程序，为了允许对这些内核服务的受控的访问，处理器提供了一条syscall指令，当用户想要请求的时候可以去执行这条指令，这将会导致一个到异常处理程序的陷阱，并调用合适的内核程序。

以下是陷阱处理的流程。

  1. 应用程序执行一次系统调用syscall
  2. 控制传递给处理程序
  3. 陷阱处理程序运行
  4. 处理程序返回到syscall之后的指令

从程序员的角度来看，系统调用和普通的函数调用时一样的。然而他们的实现很不一样，普通函数运行在用户模式中，而系统调用运行在内核模式中

故障

故障由错误情况引起，它可能能够被故障处理程序修正。当故障发生的时候，处理器将控制转移给故障处理程序，若故障处理程序能够修正这个错误，那么就会返回到引起故障的指令，否则返回到内核中的abort例程abort例程会终止引起故障的应用程序。

  1. 当前指令导致一个故障
  2. 控制传递给处理程序
  3. 故障运行程序运行
  4. 处理程序要么重新执行当前指令要么终止

一个经典的故障示例是缺页异常，当指令引用一个虚拟地址，而与该地址相对应的物理页面不在内存中，因此必须从磁盘中取出，就会发生故障。

终止

终止时不可恢复的致命错误造成的结果，通常是有一些硬件错误引起的，比如DRAM或者SRAM位被损坏时发生的就错误。种植处理程序从不将控制返回给应用程序，处理程序将控制返回给一个abort例程，该例程会终止这个应用程序

  1. 发生致命的硬件错误
  2. 传递控制给处理程序
  3. 终止处理程序云南行
  4. 处理程序返回到abort例程

进程

异常是允许操作系统内核提供进程概念的基本构造块。进程的经典定义就是一个执行中程序的实例。

我们将关注进程提供给应用程序的关键抽象

  • 一个独立的逻辑控制流，它提供一个家乡，好像我们的程序独占使用处理器
  • 一个私有的地址空间，它提供一个假象，好像我们的程序独占地使用内存系统

逻辑控制流

考虑一个运行着的三个进程的系统，处理器的一个物理控制流被分成了三个逻辑流，每个进程一个。进程A先运行一会儿，然后暂时挂起，之后进程B运行直到完成，再然后进程C运行一会儿，进程A完成，最后进程C完成。

这个例子的关键在于进程是轮流使用处理器的，每个进程执行他的流的一部分，然后暂时挂起，最后轮到其他进程。

用户模式和内核模式以及上下文切换

为了使得操作系统内核提供一个无懈可击的进程抽象，处理器必须提供一种机制，限制一个和应用可以执行的指令以及它可以的地址空间范围。处理器通常用某个寄存器中的一个模式位来提供这种功能，这个寄存器描述了进程当前享有的特权。在用户模式之下，我们无法调用内核系统中的代码，访问内核中的数据。

操作系统内核使用一种称为上下文切换的叫高层形式的异常控制流来实现多任务。比如如果一个read系统调用需要访问磁盘，内核可以选择执行上下文切换，运行另一个程序，而不是等待从磁盘到达。这样可以更高效处理任务。这种上下文切换并非只有系统调用时才会有，在许多情况下都有，例子如下

  • 时间片用完：即使程序没有等待，也会被强制切换
  • I/O操作等待：如读写文件、网络传输等
  • 主动让出CPU：程序主动调用yield()或sleep()
  • 高优先级任务就绪
  • 资源等待：如等待锁、信号量等
  • 系统调用
